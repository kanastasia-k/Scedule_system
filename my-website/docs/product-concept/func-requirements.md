## UC-1: Автоматическое составление первичного варианта расписания

**Акторы:** Сотрудник учебного отдела (авторизованный)

### Предусловия
* У пользователя есть интернет-соединение.
* Пользователь авторизован в системе как Сотрудник учебного отдела.
* В системе предварительно настроены: список аудиторий с характеристиками, расписание звонков, справочник преподавателей и учебных групп.

### Основной поток
1. Сотрудник открывает раздел «Составление расписания».
2. Сотрудник нажимает на кнопку «Загрузить учебные планы».
3. Сотрудник выбирает один или несколько Excel файлов с учебными планами и подтверждает загрузку.
4. Система считывает файлы и отображает перечень дисциплин.
5. Сотрудник открывает панель фильтров для настройки условий.
6. Система отображает доступные параметры настройки:
   * Необходимость оснащённости аудиторий оборудованием — выпадающий список с необходимым оборудованием (компьютеры, электронные доски, лабораторное оборудование) для каждой дисциплины в отдельности.
   * Учет индивидуальных учебных планов студентов — флажок да/нет для всего расписания в целом.
   * Выбор календарных ограничений (праздничных и выходных дней) — на календаре отмечаются неучебные дни для всего расписания в целом.
   * Выбор недоступных аудиторий (например, в связи с ремонтом) — отмечает галочками в выпадающем списке для всего расписания в целом.
7. Сотрудник отмечает необходимые параметры и нажимает кнопку «Сформировать расписание».
8. Система формирует первичный вариант расписания.
9. Система сохраняет расписание в БД.
10. Система отображает расписание в формате таблицы с указанием учебных групп, преподавателей и аудиторий.

### Постусловия
* **Успех:** Сотрудник получает корректный первичный вариант расписания. Все указанные в фильтрах параметры учтены алгоритмом. Расписание доступно для дальнейшей ручной корректировки, сохранения и публикации.
* **Неудача:** Отображается сообщение об ошибке. Сотрудник может изменить параметры, повторить попытку формирования расписания или обратиться в техническую поддержку.

### Исключительный поток
**Ошибка сохранения в базе данных** (Точка входа: после шага 7)
1. Система пытается сохранить изменённое расписание в базе данных.
2. Происходит ошибка на уровне хранения данных (недоступна БД, ошибка записи, превышен таймаут).
3. Система отображает сообщение: «Ошибка сохранения расписания. Повторите попытку позже или обратитесь в техническую поддержку».
4. Сотрудник может повторить попытку сохранения позже или отменить сохранение расписания.

### Альтернативные потоки
**АП 1: С указанными параметрами невозможно составить расписание** (Точка входа: после 6 шага)
1. Система не может сформировать расписание из‑за противоречивых или слишком жёстких ограничений.
2. Сотрудник получает сообщение: «Обнаружено противоречие параметров, невозможно составить расписание».
3. Система выделяет красным цветом пункты в фильтрах, которые вызвали противоречие.
4. Сотрудник изменяет параметры и возвращается к шагу 7 основного потока.

**АП 2: Ошибка чтения файла** (Точка входа: после 3 шага)
1. Система не может корректно прочитать один из файлов.
2. Система показывает сообщение: «Файл X не может быть обработан. Проверьте формат или попробуйте другой файл».
3. Сотрудник выбирает другой файл или удаляет проблемный и возвращается к шагу 3 основного потока.

### Бизнес-правила
* **BR-01:** Один преподаватель не может вести более одной дисциплины в один и тот же временной слот.
* **BR-02:** Для каждой пары должна быть назначена только одна аудитория, доступная в этот временной слот и удовлетворяющая требованиям по оборудованию.
* **BR-03:** Заранее определенное время перехода между корпусами не может занимать больше половины времени, установленного в качестве перерыва между парами расписанием звонков.
* **BR-04:** Занятия не могут назначаться в дни, помеченные как неучебные.

## UC-2: Возможность ручной корректировки расписания

**Акторы:** Сотрудник учебного отдела (авторизованный)

### Предусловия
* У пользователя есть интернет-соединение.
* Пользователь авторизован в системе как Сотрудник учебного отдела.
* Создан первичный вариант расписания.

### Основной поток
1. Сотрудник открывает сформированное расписание.
2. Сотрудник нажимает кнопку «Изменить».
3. Сотрудник изменяет параметры пары одним из способов:
   * Заменяет аудиторию и/или преподавателя (выбирает значения из выпадающих списков).
   * Перетаскивает пару на другое время или другой день.
   * Удаляет пару, нажав на неё правой кнопкой мыши и выбрав пункт «Удалить».
   * Добавляет новую пару, нажимая на свободный слот и указывая группу, дисциплину, преподавателя и аудиторию.
4. После каждого изменения система автоматически проверяет результат на:
   * Противоречия учебному плану;
   * Накладки преподавателей;
   * Накладки аудиторий.
5. При возникновении противоречий система выделяет их красным цветом.
6. Сотрудник при необходимости вносит дополнительные изменения до тех пор, пока все отмеченные противоречия не будут устранены.
7. Сотрудник нажимает кнопку «Сохранить».
8. Система сохраняет исправленный вид расписания.

### Постусловия
* **Успех:** Сотрудник получает корректный измененный вариант расписания. В расписании отсутствуют накладки и нарушения учебного плана. Расписание доступно для дальнейшей корректировки, сохранения и публикации.
* **Неудача:** Отображается сообщение об ошибке. Пользователь может изменить параметры, повторить попытку формирования или обратиться в техподдержку.

### Исключительный поток
**Ошибка сохранения в базе данных** (Точка входа: после шага 7)
1. Система пытается сохранить изменённое расписание в базе данных.
2. Происходит ошибка на уровне хранения данных.
3. Система не фиксирует изменения расписания.
4. Система отображает сообщение: «Ошибка сохранения расписания. Изменения не были сохранены. Повторите попытку позже или обратитесь в техническую поддержку».
5. Сотрудник может повторить попытку сохранения позже или отменить изменения и вернуться к последней успешно сохранённой версии.

### Альтернативный поток
**АП 1: Обнаружены неустраненные противоречия при сохранении** (Точка входа: 7 шаг)
1. При попытке сохранения обнаружены неисправленные противоречия.
2. Система отображает сообщение: «Обнаружено противоречие параметров, невозможно сохранить расписание».
3. Сотрудник изменяет параметры и возвращается к шагу 7 основного потока.

### Бизнес-правила
* **BR-01:** Один преподаватель не может вести более одной дисциплины в один и тот же временной слот.
* **BR-02:** Для каждой пары должна быть назначена только одна аудитория, доступная в этот временной слот и удовлетворяющая требованиям по оборудованию.
* **BR-03:** Заранее определенное время перехода между корпусами не может занимать больше половины времени, установленного в качестве перерыва между парами.
* **BR-04:** Занятия не могут назначаться в дни, помеченные как неучебные.

## UC-3: Интеллектуальный поиск замены при непредвиденных обстоятельствах

**Акторы:** Сотрудник учебного отдела (авторизованный), Преподаватель (авторизованный)

### Предусловия
* У пользователя есть интернет-соединение.
* Пользователь авторизован в системе.
* В расписании есть пара, для которой требуется замена.

### Основной поток
1. Пользователь открывает расписание.
2. Пользователь нажимает на пару, для которой необходимо внести изменение.
3. Пользователь выбирает, какой параметр нужно изменить (преподавателя или аудиторию).
4. Система анализирует текущие ограничения и формирует список доступных вариантов замены.
5. Система отображает список свободных преподавателей/аудиторий.
6. Пользователь выбирает один из предложенных вариантов.
7. Пользователь нажимает кнопку «Сохранить».
8. Система обновляет расписание и фиксирует изменение.

### Постусловия
* **Успех:** Пользователь получает корректную замену. Изменение отображается для всех пользователей.
* **Неудача:** Отображается сообщение об ошибке. Пользователь может повторить попытку или обратиться в техподдержку.

### Исключительный поток
**Ошибка сохранения в базе данных** (Точка входа: после шага 6)
1. Система пытается сохранить изменённое расписание в БД.
2. Происходит ошибка на уровне хранения данных.
3. Система не фиксирует изменения.
4. Система отображает сообщение: «Ошибка сохранения расписания...»
5. Сотрудник может повторить попытку или отменить изменения.

### Альтернативный поток
**АП 1: Подходящего варианта для замены нет** (Точка входа: 5 шаг)
1. Система не находит ни одного подходящего преподавателя/аудитории.
2. Система отображает сообщение: «Нет доступных вариантов замены при текущих условиях».
3. Пользователь отменяет поиск замены.

### Бизнес-правила
* **BR-01:** Один преподаватель не может вести более одной дисциплины в один и тот же временной слот.
* **BR-02:** Для каждой пары должна быть назначена только одна свободная аудитория, удовлетворяющая требованиям.
* **BR-03:** Время перехода между корпусами не может занимать больше половины времени перерыва.
* **BR-04:** Право изменять аудитории есть у Преподавателей и Сотрудников учебного отдела.
* **BR-05:** Право изменять преподавателя есть только у Сотрудников учебного отдела.

## UC-4: Возможность поиска свободных аудиторий по указанным параметрам

**Акторы:** Студент (авторизованный), Преподаватель (авторизованный)

### Предусловия
* У пользователя есть интернет-соединение.
* Пользователь авторизован в системе.

### Основной поток
1. Пользователь открывает раздел «Свободные аудитории» и нажимает кнопку «Найти свободную аудиторию».
2. Пользователь задает ограничения, выбирая: учебный корпус, наличие компьютеров, желаемое время (дата + интервал).
3. Система анализирует текущее расписание и статус аудиторий с учётом параметров.
4. Система формирует и отображает список свободных аудиторий.
5. Пользователь выбирает один из предложенных вариантов.
6. Пользователь нажимает кнопку «Забронировать аудиторию».
7. Система помечает выбранную аудиторию как занятую на указанный интервал времени.

### Постусловия
* **Успех:** Пользователь получает номер свободной аудитории. Выбранная аудитория отмечается занятой на выбранное время.
* **Неудача:** Отображается сообщение об ошибке.

### Исключительный поток
**Ошибка сохранения в базе данных** (Точка входа: после шага 6)
1. Система пытается сохранить изменения в БД.
2. Происходит ошибка записи.
3. Система не фиксирует изменения и выдает сообщение об ошибке.
4. Пользователь может повторить попытку или отменить изменения.

### Альтернативный поток
**АП 1: Подходящего варианта нет** (Точка входа: 5 шаг)
1. Система не находит ни одной подходящей аудитории.
2. Система отображает сообщение: «Нет доступных вариантов при текущих условиях».
3. Пользователь отменяет поиск замены или изменяет параметры ограничений (возврат к шагу 2).

### Бизнес-правила
* **BR-01:** Право занимать свободную аудиторию есть у Преподавателя.
* **BR-02:** Система помечает выбранную аудиторию как занятую на указанный интервал времени.
* **BR-03:** Пользователь не может забронировать аудиторию на время, выходящее за рамки расписания работы корпуса.
* **BR-04:** Право искать свободную аудиторию есть у Преподавателя и Студента.

## UC-5: Возможность просмотра расписания

**Акторы:** Авторизованный пользователь

### Предусловия
* У пользователя есть интернет-соединение.
* Пользователь авторизован в системе.

### Основной поток
1. Пользователь открывает раздел «Расписание».
2. Система отображает календарь с возможностью выбора дня.
3. Система по умолчанию отображает расписание на текущую дату.
4. Пользователь выбирает нужную дату.
5. Система отображает расписание на выбранную дату в формате таблицы (по парам, с указанием групп, дисциплин, преподавателей, аудиторий).

### Постусловия
* **Успех:** Пользователь видит актуальное расписание на выбранную дату.
* **Неудача:** Отображается сообщение об ошибке.

### Исключительный поток
**Расписание недоступно (техническая ошибка)** (Точка входа: после шага 4)
1. Система не может загрузить данные расписания (ошибка сервера, БД).
2. Система отображает сообщение: «Не удалось загрузить расписание...»
3. Пользователь может повторить попытку.

### Альтернативный поток
**АП 1: Отсутствует подключение к интернету** (Точка входа: 2 шаг)
1. Система не может загрузить данные расписания.
2. Система отображает сообщение: «Отсутствует соединение с интернетом».
3. После восстановления соединения загрузка продолжается автоматически.

### Бизнес-правила
* **BR-01:** Расписание отображается с учётом актуальных изменений и замен.

## UC-6: Формирование базовой аналитики

**Акторы:** Сотрудник учебного отдела (авторизованный), Сотрудник IT-отдела (авторизованный)

### Предусловия
* У пользователя есть интернет-соединение.
* Пользователь авторизован в системе.
* В системе есть хотя бы одно сформированное расписание.

### Основной поток
1. Сотрудник открывает раздел «Аналитика расписания».
2. Система отображает панель с типами отчётов (логи генерации, конфликты, ошибки).
3. Сотрудник выбирает нужный тип отчёта.
4. Сотрудник задаёт параметры фильтрации (диапазон дат, группы, преподаватели, корпуса).
5. Сотрудник нажимает кнопку «Сформировать отчёт».
6. Система анализирует данные и формирует отчёт:
   * Логи времени: среднее время генерации, количество попыток.
   * Конфликты: количество конфликтов по типам, процент устранённых вручную.
   * Ошибки: количество и типы технических ошибок.
7. Система отображает отчёт с возможностью экспорта (Excel/PDF).

### Постусловия
* **Успех:** Сотрудник получает готовый отчёт, доступный для просмотра и экспорта.
* **Неудача:** Отображается сообщение об ошибке.

### Исключительный поток
**Ошибка формирования отчёта** (Точка входа: после шага 5)
1. Система не может обработать данные.
2. Система отображает сообщение об ошибке формирования.
3. Сотрудник может повторить попытку или сохранить лог ошибки.

### Альтернативный поток
**АП 1: Недостаточно данных для анализа** (Точка входа: после шага 4)
1. Система обнаруживает нехватку данных за выбранный период.
2. Система отображает сообщение: «Недостаточно данных за выбранный период...»
3. Сотрудник изменяет параметры фильтрации (возврат к шагу 4).

### Бизнес-правила
* **BR-01:** Отчёты формируются только по расписаниям, созданным после внедрения системы.
* **BR-02:** Конфликты классифицируются по типам.
* **BR-03:** Успешная генерация = расписание без критических конфликтов, сохранённое сотрудником.

## UC-7: Авторизация

**Акторы:** Неавторизованный пользователь

### Предусловия
* У пользователя есть интернет-соединение и зарегистрированная учётная запись.
* Пользователь не авторизован в текущей сессии.

### Основной поток
1. Пользователь открывает страницу входа в систему.
2. Система отображает форму авторизации.
3. Пользователь вводит логин и пароль и нажимает кнопку «Войти».
4. Система проверяет корректность введённых данных.
5. При успешной проверке система создаёт авторизованную сессию пользователя.
6. Система перенаправляет пользователя на главную страницу.

### Постусловия
* **Успех:** Пользователь успешно авторизован, доступны функции в соответствии с ролью.
* **Неудача:** Пользователь остаётся неавторизованным, отображается сообщение об ошибке.

### Исключительный поток
**Ошибка доступа к БД** (Точка входа: после шага 4)
1. Система не может выполнить проверку из‑за ошибки БД.
2. Система отображает сообщение: «Сервис временно недоступен...»
3. Пользователь остаётся на странице входа.

### Альтернативные потоки
**АП 1: Неверный логин или пароль** (Точка входа: после шага 4)
1. Система не находит пользователя с такой парой логин/пароль.
2. Система отображает сообщение об ошибке.
3. Пользователь может повторить попытку ввода (возврат к шагу 3).

**АП 2: Пользователь забыл пароль** (Точка входа: шаг 2)
1. Пользователь нажимает ссылку «Забыли пароль?».
2. Система открывает форму восстановления.
3. Пользователь вводит e‑mail/логин и отправляет запрос.
4. Система подтверждает отправку инструкции на e‑mail.
5. Пользователь возвращается на страницу входа.
