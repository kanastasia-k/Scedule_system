## Категория Security (Безопасность)

*   **NFR-SEC-1: Шифрование персональных данных**
    *   **Описание:** Пароли хранятся в виде хеша (bcrypt, cost factor 12). ПДн в БД зашифрованы AES-256.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Обязательно по 152-ФЗ. За утечку данных предусмотрена юридическая ответственность.
*   **NFR-SEC-2: Блокировка аккаунта при неудачном входе**
    *   **Описание:** После 5 неудачных попыток входа подряд учётная запись блокируется на 15 минут.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Позволяет защитить аккаунт от несанкционированного входа. Стандарт безопасности.
*   **NFR-SEC-3: Аутентификация и авторизация по ролям**
    *   **Описание:** Доступ к системе только по логину/паролю. Роли: Студент, Преподаватель, Сотрудник учебного отдела, Сотрудник IT-отдела (с разграничением прав доступа к функциям).
    *   **Приоритет:** Высокий
    *   **Обоснование:** Система содержит персональные данные, которые должны быть доступны только определенным лицам.
*   **NFR-SEC-4: Защищенное соединение**
    *   **Описание:** Соединение HTTPS/TLS 1.3.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Необходимо для защиты ПДн пользователей в соответствии с 152-ФЗ. Базовый стандарт безопасности.

## Категория Performance (Производительность)

*   **NFR-PER-1: Отклик интерфейса**
    *   **Описание:** Page Load Time ≤ 2 сек (p95). Система должна выдерживать пиковую нагрузку не менее 85 RPS при времени ответа API ≤500 мс (p95).
    *   **Приоритет:** Высокий
    *   **Обоснование:** Ежедневно используется студентами и преподавателями.
*   **NFR-PER-2: Время генерации расписания**
    *   **Описание:** Генерация расписания (около 100.000 пар в семестр) ≤ 5 минут в 95% случаев. Throughput генерации ≤ 20k пар/мин.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Напрямую связано с бизнес-целью сокращения времени составления расписания с 2-4 недель до 3 дней.
*   **NFR-PER-3: Время поиска замены**
    *   **Описание:** Поиск замены аудитории/преподавателя по заданным критериям занимает менее 10 секунд.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Напрямую связано с бизнес-целью сокращения времени на поиск замены в непредвиденной ситуации.

## Категория Reliability (Надежность)

*   **NFR-REL-1: Доступность сервиса**
    *   **Описание:** Uptime системы не менее 99,9% (не более 8,7 часов в год downtime). Восстановление ≤ 1 час.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Критично в начале семестра и в момент публикации расписания, когда студенты заходят в большом количестве.
*   **NFR-REL-2: Резервное копирование**
    *   **Описание:** Ежедневно обновляемый бэкап БД.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Потеря расписания приведет к необходимости повторного составления (до 3 дней работы).

## Категория Observability (Наблюдаемость)

*   **NFR-OBS-1: Логирование операций**
    *   **Описание:** Логирование генерации, ошибок алгоритма, доступа к БД, неудачных попыток входа (время, IP). Хранение ≥ 90 дней. Метрики в дашбордах (время генерации, конфликты).
    *   **Приоритет:** Средний
    *   **Обоснование:** Необходимо для отслеживания качества работы системы и отладки алгоритма.

## Категория Regulatory (Соответствие требованиям)

*   **NFR-REG-1: Соответствие 152-ФЗ**
    *   **Описание:** Деперсонификация при аналитике. Журнал доступа к персональным данным (хранение ≥ 1 год). Хранение ПДн в РФ.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Обязательно по закону при работе с персональными данными учащихся и сотрудников.

## Категория Maintainability (Поддерживаемость)

*   **NFR-MAIN-1: Покрытие тестами**
    *   **Описание:** Unit/UI тесты критической логики ≥ 70%. Деплой через CI/CD ≤ 30 мин.
    *   **Приоритет:** Средний
    *   **Обоснование:** Гарантия стабильности при доработке алгоритма.

## Категория Usability (Удобство использования)

*   **NFR-US-1: Обучение сотрудников**
    *   **Описание:** Новый сотрудник учебного отдела осваивает систему за 1 день (≤ 10 кликов на генерацию расписания UC-1).
    *   **Приоритет:** Высокий
    *   **Обоснование:** При слишком сложном интерфейсе сотрудники вернутся к классическому способу составления.

## Категория Capacity (Вместимость)

*   **NFR-CAP-1: Масштаб пользователей**
    *   **Описание:** Поддержка 25.000 активных студентов + 1.500 преподавателей в одном вузе.
    *   **Приоритет:** Высокий
    *   **Обоснование:** Покрывает 90% вузов РФ.
*   **NFR-CAP-2: Объем хранимых данных**
    *   **Описание:** Storage: история 3 лет ≤ 500 GB.
    *   **Приоритет:** Средний
    *   **Обоснование:** Хранение данных (с индексами и логами) для вуза на 25k студентов.

## Категория Compatibility (Совместимость)

*   **NFR-COM-1: Кросс-браузерная и мобильная совместимость**
    *   **Описание:** Полная поддержка: Chrome, Яндекс Браузер, Firefox, Edge, Safari. Mobile-first responsive дизайн (320px+). Touch-friendly UI.
    *   **Приоритет:** Высокий
    *   **Обоснование:** 90% студентов смотрят расписание с телефонов ежедневно.
