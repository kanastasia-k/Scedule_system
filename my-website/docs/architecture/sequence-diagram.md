В данном разделе представлена диаграмма последовательности (Sequence Diagram), описывающая архитектурное взаимодействие компонентов системы при выполнении сценария «Интеллектуальный поиск замены при непредвиденных обстоятельствах» (UC-3).

## Диаграмма последовательности
```plantuml
@startuml
actor "Сотрудник_Преподаватель" as p
participant "Web Frontend" as f
participant "API" as api
database "Database" as db

p -> f: Клик "внести изменение"
activate f

f -> f: Вывод окна с выбором\n"Преподаватель/аудитория"
p -> f: Выбор параметра замены
f ->> api: Запрос на вывод свободных вариантов
activate api

note right of api
  Проверка бизнес-правил:
  - Конфликт времени (BR-01)
  - Занятость аудитории (BR-02)
  - Логистика корпусов (BR-03)
end note

api -> db: Запрос доступных ресурсов
activate db

alt Варианты найдены
    db --> api: Список доступных вариантов
    deactivate db
    api -->> f: Показать список доступных вариантов
    deactivate api
    
    p -> f: Выбрать подходящий вариант
    p -> f: Нажать "Сохранить"
    
    alt Данные валидны
        f ->> api: Запрос на сохранение изменений
        activate api
    
      alt Сохранение успешно
        api -> db: Сохранение изменений
        activate db
        db --> api: Успешно сохранено
        deactivate db
        api -->> f: Успешно сохранено
        deactivate api
        f -> f: Вывести уведомление\nоб успехе
        p -> f: Прочитать уведомление
      else Ошибка в API (валидация, бизнес-правила)
            note right of api
                Проверки API:
                * Актуальность сессии
                * Корректность выбранного варианта
                * Бизнес-правила (BR-01, BR-02, BR-03)
            end note
            api -->> f: Сообщение с деталями ошибки
            deactivate api
            f -> f: Показать ошибку и предложить повторить
            p -> f: Получение уведомления об ошибке
      else Ошибка при сохранении
        api -> db: Сохранение изменений
        activate db
        db --> api: Ошибка
        deactivate db
        note right of api
            Типы ошибок:
            * Нарушение бизнес-правил
            * Конфликт версий
            * Ошибка БД
            * Таймаут
            * Сетевая ошибка
        end note
        api -->> f: Сообщение с деталями ошибки
        deactivate api
        f -> f: Показать ошибку и\nпредложить повторить
        p -> f: Получение уведомления об ошибке
    end

  else Ошибка валидации на фронте
        f -> f: Показать сообщение об ошибке
        p -> f: Прочитать уведомление
  end

else Варианты не найдены
    db --> api: Нет свободных ресурсов
    deactivate db
    api -->> f: Сообщение с ошибкой:\n"Нет доступных вариантов"
    deactivate api
    f -> f: Отобразить ошибку
    p -> f: Прочитать сообщение об ошибке
end
deactivate f
@enduml
```

## Описание взаимодействия компонентов

В процессе принимают участие следующие компоненты:
*   **Акторы:** Сотрудник учебного отдела или Преподаватель.
*   **Web Frontend:** Клиентская часть приложения, отвечающая за интерфейс и первичную валидацию действий пользователя.
*   **API:** Серверная часть, реализующая основную бизнес-логику и проверки правил (Business Rules).
*   **Database:** База данных, хранящая актуальное расписание и информацию о доступности ресурсов.

### Основные сценарии (Flows)

1.  **Запрос свободных ресурсов:** Пользователь инициирует поиск замены. API обращается к БД с учетом строгих бизнес-правил (отсутствие накладок, логистика между корпусами).
2.  **Успешное сохранение (Happy Path):** Если подходящий вариант найден и выбран пользователем, фронтенд отправляет запрос на сохранение. API валидирует сессию и данные, после чего изменения успешно фиксируются в БД.
3.  **Обработка исключений:** Диаграмма предусматривает обработку различных отказов на всех уровнях:
    *   Отсутствие свободных ресурсов в БД (информационное сообщение).
    *   Ошибки клиентской валидации.
    *   Ошибки бизнес-логики на уровне API (попытка забронировать уже занятую аудиторию).
    *   Инфраструктурные ошибки (таймауты БД, сетевые сбои).
